Section 1
The Pages & Who Uses Them
There are 5 HTML pages. Customers use 3 of them. You and your team use 1. One is for customer approvals only.
🌐
index.html — Main Website
/ or /index.html
Who sees it: Everyone. This is your public storefront. Shows your product catalog (fetched live from Firebase), pricing, and an order wizard. Customers click a product and walk through a 6-step order form.
firebase-init.js products.js app.js analytics.js
👕
lp-tshirts.html — T-Shirt Landing Page
/lp-tshirts.html
Who sees it: Customers coming from ads or direct links. Same order wizard as the main site but focused only on T-shirt styles. Uses the exact same JS as index.html.
firebase-init.js products.js app.js analytics.js
📦
portal.html — Customer Order Portal
/portal.html
Who sees it: Customers who want to check on their orders. They log in with the email they used to order. They can approve or decline mockups, approve quotes, see order status, and reorder. VIP customers may see a custom catalog here.
firebase-init.js products.js orders.js production.js accounts.js catalogs.js portal.js analytics.js
🔐
admin.html — Admin Dashboard
/admin.html
Who sees it: You and your team only. Requires Firebase login (email + password). Contains everything: Kanban order board, production board, product editor, pricing, customer management, team management, commissions, analytics, and activity logs.
firebase-init.js auth.js orders.js production.js products.js accounts.js catalogs.js admin.js analytics.js
✅
approval.html — Customer Approval Page
/approval.html?id=INS-XXXXXX
Who sees it: Customers only — you send them a direct link. They can review mockups and approve or decline without logging into the portal. Each link is unique to one order.
firebase-init.js orders.js

Section 4
How an Order Becomes a Production Job
This is the critical handoff — when "approved" status is reached, the system automatically creates a production job with the right columns pre-filled based on what decoration types are on the order.
4 Ways Approval Triggers the Production Board
Way 1
Customer Approves in Portal
Customer clicks Approve on all mockups + quote in portal.html → system auto-sets status to Approved → production job created instantly
Way 2
You Drag to Approved
Drag a Kanban card to the Approved column → production job auto-created. Works for individual orders and groups.
Way 3
Quick Status Button
Click the quick-status Approved button on any Kanban card → production job auto-created
Way 4
Order Detail Modal — Save Status
Open an order detail, set status to Approved and click Save → production job created, order auto-advances to Pre-Production

How Cloud Sync Works (firebase-init.js)
Every time an admin logs in, data is reconciled between Firebase and the browser. Here's the decision logic:
1
On Page Load
App calls initCloudSync() — fetches all 6 core documents from Firestore at once
2
Timestamp Comparison
Compares the local timestamp (_ts_products) vs Firebase's updatedAt field
A
Firebase is newer → Pull down
Another team member made changes. Firebase wins. Data is written to localStorage so this browser is current.
B
Local is newer → Push up
A save happened locally before Firebase confirmed it (e.g. slow connection). Local data pushed up to Firebase to reconcile.
C
No Firebase doc yet → First-time migration
New device or new data type. Local data is pushed up to Firebase for the first time.
3
Save Flow (every change)
Any time data changes, cloudSave(docId, data) is called → writes to localStorage immediately + writes to Firebase asynchronously. If the Firebase write fails, a red banner appears at the top of the admin screen.
🌐 Public pages (index.html): Don't use initCloudSync. Instead, they sign in anonymously then fetch app_data/products directly from Firebase on every page load so visitors always see the latest catalog.
Firebase Storage — Photo Files
Product color photos are too large to store inside Firestore documents. They go to Firebase Storage instead, and only the URL is stored in the product data.

Section 6
Admin Login & Team Access
Who can log in, what they can see, and how access is controlled.
Login Flow (admin.html)
  • 1
    Team member visits admin.html and enters their email + password
  • 2
    Firebase Auth verifies credentials
  • 3
    System checks their profile in the admins Firestore collection
  • 4
    Result depends on status:
    ✓ approved = true → full admin access
    ⏳ status = pending → "awaiting approval" screen
    ✗ status = rejected → "access denied" screen
    ? not found → signed out
  • 5
    On each login: last_logins collection updated + activity log written
Roles & Permissions
super_admin (Blake)
Only one. Set by hardcoded email. Can see all dashboard sections. Manages team access, approves/rejects employees, sees all data. No restrictions.
admin (Team Members)
Must be invited or approved by Blake. What they see on their dashboard can be toggled per person (Alerts, KPI, Analytics, Due Dates). Same data access otherwise.
Customer (Portal Users)
Email-based login, no Firebase Auth. Can only see their own orders. No access to any admin data. VIP customers may see a custom catalog.
Dashboard Section Toggles (per team member)
In Admin → Team page, you can turn on/off each dashboard section for each team member. Super admin always sees everything.

Section 7
JavaScript Files — What Each One Does
Each file has a single job. They're loaded on the pages that need them.
📡 firebase-init.js
cloudSave(docId, data)
Writes data to Firebase + updates localStorage timestamp
cloudLoad(docId, cb)
Reads a single document from Firebase
initCloudSync(onComplete)
On startup: syncs all 6 data documents between browser and Firebase
uploadToStorageWithProgress(blob, path, onProgress)
Uploads a photo to Firebase Storage with real-time % progress callback
🔐 auth.js
initAuth(onAuthed, onUnauthed)
Firebase Auth listener — boots or rejects admin on login
authAddTeamMember(name, email)
Creates Firebase user + admin profile, sends password reset
authApproveEmployee / authRevokeEmployee
Moves access requests to approved admins or revokes access
logActivity(action, type, id, detail)
Writes an entry to the activity audit log in Firestore
📋 orders.js
createOrder(wizardData)
Builds a new order object from wizard data, saves to Firebase
updateOrder(id, changes)
Patches an existing order and re-saves
getOrders() / saveOrders(orders)
Read/write the full orders array from localStorage + Firebase
getStatusInfo(id) / getStatusColumn(id)
Looks up label and color for any order status ID
🏭 production.js
ensureProductionJob(order)
Creates a production board job from an order. Handles groups, merges, and initializes the right columns based on decoration types.
getMasterStatus(job)
Returns pre-production / in-production / done based on column progress
getJobUrgency(job)
Returns red/yellow/null based on deadline proximity
updateProductionJob / removeProductionJob
Modify or remove production jobs, handles group membership
👕 products.js
getProducts() / saveProducts(products)
Read/write product catalog from localStorage + Firebase
calcPricePerPiece(blankCost, decoId, qty)
Calculates sell price using the multiplier pricing grid
DEFAULT_PRODUCTS
Fallback product list if Firebase is unreachable
getPricingMetrics() / savePricingMetrics()
The multiplier grid tables for all decoration types
🧙 app.js
renderProducts()
Builds the product grid on the public website from live Firebase data
openWizard(productId)
Opens the 6-step order wizard for a product
wizardNext() / wizardBack()
Navigate wizard steps, saves abandon checkpoint on each advance
placeOrders()
Final step — creates orders in Firebase, clears abandon record
🏠 portal.js
portalLogin() / portalLogout()
Email-based session for customers (no Firebase Auth needed)
approveMockup(orderId, mockupId)
Marks a mockup as approved, checks if all done → auto-advances order
declineMockup(orderId, mockupId, reason)
Records decline reason, sets order back to revisions-needed
approveQuote(orderId)
Marks quote approved, checks if all done → auto-advances to Approved
_checkPortalApprovals(orderId)
After any approval, checks if everything is done and fires production push
📈 analytics.js
Auto-runs on all public pages
Signs in anonymously → records a page view in Firestore app_data/analytics. Tracks: total views, unique sessions (per browser), daily breakdown, and per-page counts. Prunes data older than 90 days automatically.
🏷️ accounts.js
Customer portal account management
Handles VIP customer accounts (separate from admin Firebase Auth). Password hashing, account creation, session management for repeat customers who want a saved login.
📚 catalogs.js
VIP custom catalog management
Admin can create a custom product catalog for specific customer emails with special pricing. When that customer logs into the portal, they see their curated catalog alongside their orders.
